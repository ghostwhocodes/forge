The `recommended_next_action` field MUST be exactly one of:

- `continue` — the latest slice succeeded, more meaningful work likely remains, the next slice is still bounded, no design-boundary stop is active, and the policy still allows another unattended round
- `complete` — the latest accepted slice leaves no worthwhile unattended next round, including cases where valid remaining work should be handled by contract simplification or human design review instead of another patch
- `escalate` — the round budget is exhausted, the risk changed materially, or human judgment is now required before any safe continuation

Decision guidance:

- return `continue` only when there is still material progress, a bounded next slice remains, and the rabbit-hole report does not recommend stopping for redesign
- return `complete` when the accepted review process or rabbit-hole report indicates that no worthwhile unattended slice remains or that the run has reached a design boundary but is otherwise safe to hand to the final human summary/signoff path
- return `escalate` when the current accepted round already equals `max_rounds` and more work still remains, a new high-severity issue changes the risk profile, or the child outputs are too inconsistent to hand off safely

Use the rabbit-hole report to decide whether the loop is still sound. If that report says the component has drifted beyond its intended purpose, set `rabbit_hole_detected=true`, set `design_boundary_hit=true`, choose `recommended_human_follow_up` accordingly, and do not return `continue`.

Keep the status artifact operational. Prefer explicit telemetry and short reasons over vague prose.
