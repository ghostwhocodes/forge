Also write warnings JSON as cumulative loop telemetry. Preserve and update prior history when re-entry context provides earlier warnings. Include at least:

- max_review_cycles
- blocking_threshold
- latest_blocking_findings
- current_blocking_finding_ids
- seen_blocking_finding_ids
- resolved_blocking_finding_ids
- new_findings_after_rework
- reopened_findings
- remediation_happened
- escalated_due_to_cycle_budget
- warning_messages

When computing these fields:
- current_blocking_finding_ids should list the stable ids of currently open blocking findings
- seen_blocking_finding_ids should accumulate every blocking finding id seen in any prior round
- resolved_blocking_finding_ids should accumulate ids that were blocking in an earlier round and are not blocking now
- new_findings_after_rework should count current blocking ids that were not present in the prior seen set
- reopened_findings should count current blocking ids that were previously in resolved_blocking_finding_ids

Keep warning_messages concrete and operational. Frame them as stability or convergence signals, not as a hard diagnosis of the request.
